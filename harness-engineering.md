# Harness Engineering: 에이전트 퍼스트 시대에서 Codex 활용하기

> **저자:** Ryan Lopopolo, Member of the Technical Staff, OpenAI
> **날짜:** 2026-02-11
> **원문:** https://openai.com/index/harness-engineering/

---

## 개요

지난 5개월간 우리 팀은 하나의 실험을 진행했다: **수동으로 작성한 코드 0줄**로 소프트웨어 제품의 내부 베타를 구축하고 출시하는 것.

이 제품에는 내부 일일 사용자와 외부 알파 테스터가 있다. 배포되고, 디플로이되고, 장애가 나고, 수정된다. 다른 점은 모든 코드 — 애플리케이션 로직, 테스트, CI 구성, 문서, 관측성, 내부 도구 — 가 Codex에 의해 작성되었다는 것이다. 수작업으로 코드를 작성했을 때 대비 **약 1/10의 시간**에 구축한 것으로 추정된다.

**사람은 조종하고, 에이전트가 실행한다.**

우리는 의도적으로 이 제약을 선택했다. 엔지니어링 속도를 수십 배 높이는 데 필요한 것을 만들기 위해서였다. 결과적으로 약 100만 줄의 코드를 몇 주 만에 출시해야 했다. 이를 위해 소프트웨어 엔지니어링 팀의 핵심 업무가 더 이상 코드를 작성하는 것이 아니라 **환경을 설계하고, 의도를 명세하고, Codex 에이전트가 신뢰할 수 있는 작업을 수행할 수 있는 피드백 루프를 구축하는 것**으로 바뀌면 무엇이 달라지는지 이해해야 했다.

이 글은 에이전트 팀과 함께 새로운 제품을 구축하면서 배운 것에 대한 이야기다 — 무엇이 깨졌고, 무엇이 복리로 쌓였으며, 진정으로 희소한 자원인 **사람의 시간과 주의력**을 어떻게 극대화할 것인가.

## 빈 Git 리포지토리에서 시작

첫 번째 커밋은 2025년 8월 말에 빈 리포지토리에 랜딩되었다.

초기 스캐폴드 — 리포지토리 구조, CI 구성, 포매팅 규칙, 패키지 매니저 설정, 애플리케이션 프레임워크 — 는 기존 템플릿 세트를 참고하여 GPT-5를 사용하는 Codex CLI로 생성했다. 에이전트에게 리포지토리에서 어떻게 작업해야 하는지 안내하는 초기 AGENTS.md 파일조차 Codex가 작성했다.

시스템을 고정시킬 사전 존재하는 사람이 작성한 코드는 없었다. 처음부터 리포지토리는 에이전트에 의해 형성되었다.

5개월 후, 리포지토리에는 애플리케이션 로직, 인프라, 도구, 문서, 내부 개발자 유틸리티에 걸쳐 **약 100만 줄의 코드**가 포함되어 있다. 이 기간 동안 엔지니어 3명의 소규모 팀이 Codex를 구동하여 약 **1,500개의 PR**을 오픈하고 머지했다. 이는 **엔지니어당 하루 평균 3.5 PR**의 처리량에 해당하며, 놀랍게도 팀이 현재 7명의 엔지니어로 성장하면서 처리량이 증가했다. 중요한 것은 이것이 산출물을 위한 산출물이 아니었다는 점이다: 이 제품은 내부적으로 수백 명의 사용자가 사용했고, 일일 내부 파워 유저도 포함되어 있다.

개발 과정 전체에서 사람은 직접 코드를 기여한 적이 없다. 이것이 팀의 핵심 철학이 되었다: **수동으로 작성한 코드 없음**.

## 엔지니어의 역할 재정의

사람이 직접 코딩하지 않게 되면서, 시스템·스캐폴딩·레버리지에 초점을 맞춘 다른 종류의 엔지니어링 작업이 등장했다.

초기 진행은 예상보다 느렸다. Codex가 무능해서가 아니라 **환경이 불충분하게 명세**되어 있었기 때문이다. 에이전트에게 고수준 목표를 향해 진행하는 데 필요한 도구, 추상화, 내부 구조가 부족했다. 우리 엔지니어링 팀의 주요 업무는 **에이전트가 유용한 작업을 수행할 수 있도록 만드는 것**이 되었다.

실제로 이것은 깊이 우선(depth-first)으로 작업하는 것을 의미했다: 큰 목표를 작은 빌딩 블록(설계, 코드, 리뷰, 테스트 등)으로 분해하고, 에이전트에게 해당 블록을 구성하도록 프롬프트하고, 이를 사용하여 더 복잡한 작업을 해제하는 것. 무언가 실패하면 수정은 거의 "더 열심히 시도하기"가 아니었다. Codex가 작업을 수행하도록 하는 것이 진행의 유일한 방법이었기 때문에, 사람 엔지니어는 항상 작업에 개입하여 물었다: **"어떤 능력이 부족하고, 에이전트가 읽고 강제할 수 있도록 어떻게 만들 것인가?"**

사람은 거의 전적으로 프롬프트를 통해 시스템과 상호작용한다: 엔지니어가 작업을 설명하고, 에이전트를 실행하고, PR을 열도록 한다. PR을 완성까지 이끌기 위해, Codex에게 자신의 변경사항을 로컬에서 리뷰하고, 로컬과 클라우드 모두에서 추가 에이전트 리뷰를 요청하고, 사람 또는 에이전트 피드백에 응답하고, 모든 에이전트 리뷰어가 만족할 때까지 루프에서 반복하도록 지시한다.

사람이 PR을 리뷰할 수 있지만 필수는 아니다. 시간이 지나면서 거의 모든 리뷰 노력을 **에이전트-대-에이전트**로 처리하도록 전환했다.

## 애플리케이션 가독성(Legibility) 높이기

코드 처리량이 증가하면서 병목은 **사람 QA 용량**이 되었다. 고정된 제약이 사람의 시간과 주의력이었기 때문에, 앱 UI, 로그, 앱 메트릭 같은 것들을 Codex가 직접 읽을 수 있도록 만들어 에이전트에 더 많은 능력을 추가하는 작업을 했다.

예를 들어:

- **앱을 Git worktree별로 부팅 가능**하게 만들어, Codex가 변경마다 하나의 인스턴스를 실행하고 구동할 수 있게 했다
- **Chrome DevTools Protocol**을 에이전트 런타임에 연결하고, DOM 스냅샷·스크린샷·네비게이션 작업을 위한 스킬을 생성했다
- 이를 통해 Codex가 버그를 재현하고, 수정을 검증하고, UI 동작을 직접 추론할 수 있게 되었다

관측성 도구에도 동일하게 적용했다:

- 로그, 메트릭, 트레이스가 **worktree별 임시 로컬 관측성 스택**을 통해 Codex에 노출
- 에이전트가 **LogQL로 로그**, **PromQL로 메트릭**을 쿼리할 수 있음
- "서비스 시작이 800ms 이내에 완료되도록 보장하라" 또는 "이 4개 핵심 사용자 여정의 어떤 스팬도 2초를 초과하지 않아야 한다" 같은 프롬프트가 실행 가능해짐

단일 Codex 실행이 **6시간 이상** 하나의 작업에 매달리는 것을 정기적으로 관찰한다 (종종 사람이 자고 있는 동안).

## 리포지토리 지식을 시스템 오브 레코드로 만들기

컨텍스트 관리는 에이전트가 크고 복잡한 작업에서 효과적이 되도록 하는 가장 큰 도전 중 하나다. 가장 일찍 배운 교훈은 단순했다: **에이전트에게 지도를 줘라, 1,000페이지 설명서 말고.**

### "하나의 거대한 AGENTS.md" 접근법의 실패

- **컨텍스트는 희소한 자원이다.** 거대한 지침 파일은 작업, 코드, 관련 문서를 밀어낸다 — 에이전트가 핵심 제약을 놓치거나 잘못된 것을 최적화하기 시작한다.
- **너무 많은 가이던스는 비-가이던스가 된다.** 모든 것이 "중요"하면 아무것도 중요하지 않다. 에이전트가 의도적으로 탐색하는 대신 로컬에서 패턴 매칭을 하게 된다.
- **즉시 노후화된다.** 단일 매뉴얼은 낡은 규칙의 묘지가 된다. 에이전트는 무엇이 아직 유효한지 구분할 수 없고, 사람은 유지보수를 멈추고, 파일은 조용히 위험 요소가 된다.
- **검증이 어렵다.** 단일 블롭은 기계적 검사(커버리지, 신선도, 소유권, 교차 링크)에 적합하지 않아 드리프트가 불가피하다.

### 해결: AGENTS.md는 목차, 지식은 구조화된 docs/에 배치

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

- 설계 문서는 검증 상태와 에이전트 퍼스트 운영 원칙을 정의하는 핵심 신념 세트를 포함하여 목록화되고 인덱싱됨
- 아키텍처 문서는 도메인과 패키지 레이어링의 최상위 맵을 제공
- 품질 문서는 각 제품 도메인과 아키텍처 레이어에 등급을 매기고 시간에 따른 갭을 추적
- **계획은 일급 아티팩트로 취급.** 소규모 변경에는 경량 계획, 복잡한 작업에는 진행 및 의사결정 로그가 포함된 실행 계획(exec-plans)이 리포지토리에 체크인
- 활성 계획, 완료된 계획, 알려진 기술 부채 모두 버전 관리되고 함께 배치

이를 통해 **점진적 공개(progressive disclosure)** 가 가능해진다: 에이전트는 작고 안정적인 진입점에서 시작하고, 다음에 어디를 봐야 하는지 학습하며, 처음부터 압도되지 않는다.

이는 기계적으로 강제된다:
- 전용 린터와 CI 작업이 지식 베이스가 최신이고, 교차 링크되고, 올바르게 구조화되어 있는지 검증
- 정기적인 **"doc-gardening" 에이전트**가 실제 코드 동작을 반영하지 않는 낡거나 폐기된 문서를 스캔하고 수정 PR을 오픈

## 에이전트 가독성이 목표

리포지토리가 전적으로 에이전트에 의해 생성되었기 때문에, **Codex의 가독성에 최적화**되어 있다. 팀이 신입 엔지니어를 위해 코드 탐색성을 개선하는 것처럼, 우리 엔지니어의 목표는 에이전트가 리포지토리 자체에서 직접 전체 비즈니스 도메인에 대해 추론할 수 있도록 만드는 것이었다.

에이전트 관점에서, **실행 중 컨텍스트에서 접근할 수 없는 것은 사실상 존재하지 않는다.** Google Docs, 채팅 스레드, 사람 머릿속의 지식은 시스템에 접근 불가능하다. 리포지토리 로컬의 버전 관리된 아티팩트(코드, 마크다운, 스키마, 실행 계획)만이 에이전트가 볼 수 있는 전부다.

> 팀의 아키텍처 패턴을 정렬한 Slack 논의? 에이전트가 발견할 수 없다면, 3개월 후 합류하는 신입에게 알려지지 않은 것과 같은 방식으로 불가독하다.

핵심 트레이드오프:
- 레포 내에서 완전히 내재화하고 추론할 수 있는 의존성과 추상화를 선호
- **"지루한" 기술**이 에이전트가 모델링하기 쉬운 경향 — 조합성, API 안정성, 훈련 세트에서의 표현 때문
- 외부 라이브러리의 불투명한 동작을 우회하는 것보다, 에이전트가 기능의 하위 집합을 **재구현하는 것이 더 저렴한 경우**도 있음
  - 예: 일반적인 `p-limit` 스타일 패키지 대신, OpenTelemetry 계측과 긴밀히 통합된 자체 map-with-concurrency 헬퍼를 구현 (100% 테스트 커버리지)

## 아키텍처와 맛(Taste) 강제하기

문서만으로는 완전히 에이전트가 생성한 코드베이스를 일관성 있게 유지할 수 없다. **구현을 세세하게 관리하지 않고 불변 조건을 강제**함으로써, 기반을 훼손하지 않으면서 빠르게 출시할 수 있게 했다.

예를 들어, Codex에게 경계에서 데이터 형태를 파싱하도록 요구하지만, 그것이 어떻게 이루어지는지에 대해서는 규정하지 않는다 (모델은 Zod를 좋아하는 것 같지만, 우리가 그 특정 라이브러리를 지정하지는 않았다).

### 계층 도메인 아키텍처

각 비즈니스 도메인은 고정된 레이어 세트로 나뉘며, 엄격하게 검증된 의존성 방향과 제한된 허용 가능한 엣지 세트를 가진다:

```
Types → Config → Repo → Service → Runtime → UI
```

교차 관심사(auth, connectors, telemetry, feature flags)는 단일 명시적 인터페이스인 **Providers**를 통해 진입한다. 그 외는 허용되지 않으며 기계적으로 강제된다.

이것은 보통 수백 명의 엔지니어가 있을 때까지 미루는 종류의 아키텍처다. 코딩 에이전트와 함께라면, **초기 전제 조건**이다: 제약이 쇠퇴나 아키텍처 드리프트 없이 속도를 가능하게 하는 것이다.

### 강제 수단

- **커스텀 린터와 구조적 테스트** + 소규모 "맛 불변조건(taste invariants)" 세트
- 구조화된 로깅, 스키마와 타입의 네이밍 규칙, 파일 크기 제한, 플랫폼별 신뢰성 요구사항을 정적으로 강제
- 커스텀 린트이기 때문에, **에러 메시지에 수정 지침을 작성하여 에이전트 컨텍스트에 주입**

> 사람 중심 워크플로우에서 이런 규칙은 지나치게 세밀하거나 제약적으로 느껴질 수 있다. 에이전트와 함께라면 이것은 **승수(multiplier)** 가 된다: 한 번 인코딩되면 모든 곳에 동시에 적용된다.

동시에, 제약이 중요한 곳과 중요하지 않은 곳에 대해 명시적이다. 이는 대규모 엔지니어링 플랫폼 조직을 이끄는 것과 유사하다: **중앙에서 경계를 강제하고, 로컬에서는 자율성을 허용한다.**

결과 코드가 항상 사람의 스타일적 선호와 일치하지 않을 수 있으며, **그것은 괜찮다.** 출력이 정확하고, 유지보수 가능하고, 미래 에이전트 실행에 가독적이면 기준을 충족한다.

사람의 맛은 시스템에 지속적으로 피드백된다. 리뷰 코멘트, 리팩토링 PR, 사용자 대면 버그는 문서 업데이트로 캡처되거나 도구에 직접 인코딩된다. **문서가 부족하면, 규칙을 코드로 승격시킨다.**

---

## 처리량이 머지 철학을 바꾼다

Codex의 처리량이 증가하면서, 많은 전통적인 엔지니어링 규범이 **비생산적**이 되었다.

리포지토리는 최소한의 블로킹 머지 게이트로 운영된다. PR은 수명이 짧다. 테스트 플레이크(flake)는 진행을 무기한 차단하기보다 후속 실행으로 해결되는 경우가 많다. 에이전트 처리량이 사람의 주의력을 훨씬 초과하는 시스템에서는 **교정은 저렴하고, 기다리는 것은 비싸다.**

저처리량 환경에서는 무책임할 것이다. 여기서는 종종 올바른 트레이드오프다.

---

## "에이전트 생성"이 실제로 의미하는 것

코드베이스가 Codex 에이전트에 의해 생성되었다고 할 때, 코드베이스의 **모든 것**을 의미한다.

에이전트가 생산하는 것:
- 제품 코드와 테스트
- CI 구성과 릴리스 도구
- 내부 개발자 도구
- 문서와 설계 이력
- 평가 하네스
- 리뷰 코멘트와 응답
- 리포지토리 자체를 관리하는 스크립트
- 프로덕션 대시보드 정의 파일

사람은 항상 루프에 있지만, **이전과 다른 추상화 레이어**에서 작업한다. 우리는 작업의 우선순위를 정하고, 사용자 피드백을 수용 기준으로 번역하고, 결과를 검증한다. 에이전트가 어려움을 겪으면 신호로 취급한다: 무엇이 빠져 있는지 — 도구, 가드레일, 문서 — 를 파악하고 리포지토리에 피드백하되, 항상 **Codex 자체가 수정을 작성**하도록 한다.

---

## 자율성의 단계적 확대

개발 루프의 더 많은 부분이 시스템에 직접 인코딩되면서 — 테스트, 검증, 리뷰, 피드백 처리, 복구 — 리포지토리는 최근 Codex가 **새 기능을 엔드-투-엔드로 구동**할 수 있는 의미 있는 임계점을 넘었다.

단일 프롬프트가 주어지면, 에이전트는 이제 다음을 수행할 수 있다:

1. 코드베이스의 현재 상태 검증
2. 보고된 버그 재현
3. 실패를 보여주는 영상 녹화
4. 수정 구현
5. 애플리케이션을 구동하여 수정 검증
6. 해결을 보여주는 두 번째 영상 녹화
7. PR 오픈
8. 에이전트와 사람 피드백에 응답
9. 빌드 실패 감지 및 교정
10. 판단이 필요할 때만 사람에게 에스컬레이션
11. 변경 머지

이 동작은 이 리포지토리의 특정 구조와 도구에 크게 의존하며, 유사한 투자 없이 일반화되지 않는다고 가정해야 한다 — 적어도 아직은.

---

## 엔트로피와 가비지 컬렉션

완전한 에이전트 자율성은 새로운 문제도 도입한다. Codex는 리포지토리에 이미 존재하는 패턴을 복제한다 — **고르지 않거나 최적이 아닌 패턴까지.** 시간이 지나면 이것은 불가피하게 드리프트로 이어진다.

초기에 사람이 수동으로 이를 해결했다. 우리 팀은 매주 금요일(주의 20%)을 **"AI 슬롭" 정리**에 사용했다. 당연히 이는 확장되지 않았다.

대신, 우리가 **"골든 원칙(golden principles)"** 이라 부르는 것을 리포지토리에 직접 인코딩하고 정기적인 정리 프로세스를 구축했다:

- 손으로 만든 헬퍼보다 **공유 유틸리티 패키지**를 선호하여 불변 조건을 중앙화
- 데이터를 "YOLO 스타일"로 탐지하지 않음 — 경계를 검증하거나 타입이 지정된 SDK에 의존하여 에이전트가 추측된 형태 위에 실수로 빌드하지 않도록 함
- 정기적 주기로 백그라운드 Codex 작업 세트가 **위반을 스캔하고, 품질 등급을 갱신하고, 타겟 리팩토링 PR을 오픈**
- 대부분 1분 이내에 리뷰 가능하며 자동 머지됨

이것은 **가비지 컬렉션**처럼 기능한다. 기술 부채는 고금리 대출과 같다: 작은 증분으로 지속적으로 갚는 것이 복리로 쌓이게 두고 고통스러운 폭발적 처리를 하는 것보다 거의 항상 낫다. 사람의 맛은 한 번 캡처되고, 그 다음 모든 코드 라인에 지속적으로 강제된다. 이를 통해 나쁜 패턴을 며칠이나 몇 주 동안 퍼지게 두는 대신 **매일** 잡아서 해결할 수 있다.

## 아직 배우고 있는 것

이 전략은 지금까지 OpenAI에서의 내부 출시와 도입까지 잘 작동했다. 실제 사용자를 위한 실제 제품을 구축하는 것이 투자를 현실에 고정시키고 장기적 유지보수성을 안내하는 데 도움이 되었다.

아직 모르는 것:
- 완전히 에이전트가 생성한 시스템에서 **아키텍처 일관성이 수년에 걸쳐 어떻게 진화**하는지
- **사람의 판단이 가장 큰 레버리지를 추가하는 곳**이 어디인지, 그리고 그 판단을 복리로 쌓이도록 인코딩하는 방법
- 모델이 시간이 지남에 따라 더 능력이 향상되면서 이 시스템이 어떻게 진화할지

분명해진 것: 소프트웨어 구축은 여전히 규율을 요구하지만, **규율은 코드가 아닌 스캐폴딩에서 나타난다.** 코드베이스를 일관성 있게 유지하는 도구, 추상화, 피드백 루프가 점점 더 중요해지고 있다.

우리의 가장 어려운 도전은 이제 에이전트가 목표를 달성하도록 돕는 **환경, 피드백 루프, 제어 시스템을 설계하는 것**에 집중되어 있다: 대규모로 복잡하고 신뢰할 수 있는 소프트웨어를 구축하고 유지하는 것.

Codex와 같은 에이전트가 소프트웨어 생명주기의 더 많은 부분을 담당함에 따라, 이 질문들은 더욱 중요해질 것이다. 우리가 초기 교훈의 일부를 공유하는 것이 여러분이 노력을 어디에 투자할지 판단하는 데 도움이 되기를 바란다 — 그래서 그냥 만들 수 있도록.
