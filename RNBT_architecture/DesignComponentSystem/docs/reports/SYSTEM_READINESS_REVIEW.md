# 컴포넌트 시스템 공식 도입 준비 검토

검토일: 2026-03-25

---

## 검토 목적

두 가지를 판단한다.

1. **구조 전략**: 대량의 컴포넌트를 확보하려는 전략적 목적으로, 현재의 Mixin/Design 구조가 합당한가
2. **명세 도구**: 팀에게 "이 템플릿 기반으로 명세를 작성해달라"고 공식 요청할 수 있는 수준인가

---

## 1. 구조 전략 평가

### 핵심 주장

> N개 디자인 + M개 기능 = N + M 관리량으로 N × M 이상의 조합을 생산한다.

이 주장이 성립하려면 세 가지 조건이 필요하다.

### 조건별 검증

| 조건 | 내용 | 판정 |
|------|------|------|
| **독립 교환** | 기능과 디자인이 서로를 몰라도 결합 가능한가 | 성립 |
| **조립 최소성** | 조합 시 추가 코드가 최소인가 | 성립 |
| **무수정 확장** | 새 조합을 만들 때 기존 코드를 수정할 필요가 없는가 | 성립 |

**독립 교환 — 성립하는 근거:**

- 선택자 계약(cssSelectors/datasetAttrs)이 인터페이스 역할을 한다
- 디자인(HTML)이 약속된 선택자를 제공하면, 어떤 시각 구조든 같은 기능 모듈이 동작한다
- 실제로 SystemInfo가 3개 디자인 변형(bar, card, minimal)을 하나의 조립 코드로 공유하고 있다
- 기능 모듈은 HTML 구조를 모르고, 디자인은 기능 내부를 모른다

**조립 최소성 — 성립하는 근거:**

- 조립 코드(register.js)는 3단계 패턴이 고정되어 있다: (1) 기능 적용 → (2) 구독 연결 → (3) 이벤트 매핑
- 각 단계에서 변하는 것은 선택자 문자열과 topic 이름뿐이다
- 실제 조립 코드 크기가 30~50줄로 제한된다
- 모범답안 10개의 register.js를 대조한 결과, 모두 이 3단계를 벗어나지 않는다

**무수정 확장 — 성립하는 근거:**

- 새 컴포넌트 추가 = 새 HTML/CSS + 새 register.js. 기존 기능 모듈 무수정
- 새 기능 모듈 추가 = 새 apply 함수. 기존 컴포넌트 무수정
- 설계 문서의 "확장 경로" 섹션이 이를 명시하고, 실제 구조가 이를 위반하는 지점이 없다

### 전략 판정

**이 구조는 대량 컴포넌트 확보 전략으로서 합당하다.**

- 기능과 시각의 직교성이 실제로 보장되어 있다
- 새 디자인이 들어올 때 기능 코드를 다시 쓸 필요가 없다
- 새 기능이 필요할 때 기존 디자인을 수정할 필요가 없다
- "어떤 프로젝트가 올지 모르는" 플랫폼 상황에서 기능과 시각을 미리 분리해두는 것은 올바른 선택이다

### 현재 커버리지의 한계

전략이 성립하는 범위에 한계가 있다.

| 현재 커버하는 영역 | 아직 없는 영역 |
|-------------------|---------------|
| 데이터 표시 (텍스트, 목록, 차트, 테이블, 히트맵) | 입력/폼 (Form) |
| 상태 반영 (DOM, 3D 메시) | 네비게이션 |
| 시점 전환 (카메라) | 레이아웃 제어 |

이것은 **구조의 문제가 아니라 커버리지의 문제**이며, 구조 자체는 확장을 수용할 수 있다. 기능 모듈 범용성 검토([MIXIN_REVIEW.md](MIXIN_REVIEW.md))에서 이 한계를 이미 식별하고 있다.

## 2. 명세 템플릿/모범답안 평가

"이 템플릿 기반으로 명세를 작성해달라"고 팀에게 공식 요청할 수 있는 수준인지를 **일관성**, **합리성**, **완결성** 세 축으로 검토한다.

### 검토 대상

| 종류 | 파일 | 수량 |
|------|------|------|
| 템플릿 | [MIXIN_SPEC_TEMPLATE.md](../specs/MIXIN_SPEC_TEMPLATE.md) | 1개 |
| 모범답안 | MIXIN_SPEC_EXAMPLE_*.md | 14개 |
| 설계 근거 | [COMPONENT_SYSTEM_DESIGN.md](../architecture/COMPONENT_SYSTEM_DESIGN.md), [FUNCTION_PATTERN.md](../architecture/FUNCTION_PATTERN.md) | 2개 |

### A. 일관성 — 높음

**섹션 구조:**

14개 모범답안 모두 동일한 6개 섹션을 따른다.

| 섹션 | 내용 |
|------|------|
| 1. 기능 정의 | 왜 존재하는가 (목적 + 기능 + 기존 Mixin과의 관계) |
| 2. 인터페이스 | 무엇을 받는가 (cssSelectors, datasetAttrs, 기타 옵션) |
| 3. renderData 기대 데이터 | 어떤 데이터를 기대하는가 |
| 4. 주입 네임스페이스 | 무엇을 주입하는가 |
| 5. destroy 범위 | 무엇을 정리하는가 |
| 6. 사용 예시 | 어떻게 쓰는가 |

3D Mixin은 cssSelectors/datasetAttrs 대신 "해당 없음"을 명시하고 기타 옵션으로 대체한다. 이 처리 방식이 모범답안 간에 통일되어 있다.

**목적/기능 서술:**

| Mixin | 목적 | 추상화 수준 |
|-------|------|------------|
| FieldRenderMixin | 데이터를 보여준다 | 추상 행위 |
| ListRenderMixin | 데이터를 보여준다 (`itemKey` 옵션 시 개별 항목 상태 변경 포함) | 추상 행위 |
| EChartsMixin | 데이터를 보여준다 | 추상 행위 |
| TabulatorMixin | 데이터를 보여준다 | 추상 행위 |
| HeatmapJsMixin | 데이터를 보여준다 | 추상 행위 |
| MeshStateMixin | 데이터를 보여준다 (3D 메시의 시각 상태 변경) | 추상 행위 |
| MeshHighlightMixin | 선택한 대상을 시각적으로 강조한다 | 추상 행위 |
| MeshVisibilityMixin | 3D 장면의 특정 부분을 선택적으로 보여준다 | 추상 행위 |
| CameraFocusMixin | 보는 위치를 전환한다 | 추상 행위 |
| AnimationMixin | 3D 모델의 동작을 재현한다 | 추상 행위 |
| ClippingPlaneMixin | 3D 모델의 내부를 보여준다 | 추상 행위 |
| ShadowPopupMixin | 콘텐츠를 별도 레이어에 표시한다 | 추상 행위 |
| 3DShadowPopupMixin | 콘텐츠를 별도 레이어에 표시한다 (3D 환경) | 추상 행위 |
| TreeRenderMixin | 데이터를 보여준다 (트리 구조 렌더링) | 추상 행위 |

목적은 모두 "~한다" 형태로, 수단을 포함하지 않는다. 기능은 모두 목적 + 수단을 결합한 한 문장이다.

**기존 Mixin과의 관계:**

모든 모범답안이 "목적이 같은 기존 Mixin"과 "기능의 차이"를 명시한다. 팀 구성원이 새 기능 모듈을 제안할 때 기존과의 중복을 자각하도록 유도한다.

### B. 합리성 — 높음

**템플릿 섹션 배치:**

존재 이유 → 계약 → 소비 → 주입 → 정리 → 사용. 이 순서는 기능 모듈을 이해하고 구현하는 자연스러운 흐름을 따른다.

**"기능의 정의" 프레임워크:**

[COMPONENT_SYSTEM_DESIGN.md](../architecture/COMPONENT_SYSTEM_DESIGN.md)에서 정의한 "기능 = 목적 + 수단" 프레임워크가 [FUNCTION_PATTERN.md](../architecture/FUNCTION_PATTERN.md)에서 순수 함수(map, filter, debounce, throttle)로 검증되어 있다.

팀이 "왜 이렇게 나누는가?"라고 질문했을 때 즉시 답변할 수 있는 근거 문서가 있다.

- "목적이 같고 기능이 다르면 별도" (debounce ≠ throttle, FieldRender ≠ ListRender)
- "기능이 같으면 하나로 충분" (map은 하나, FieldRender는 하나)

이 원칙이 모범답안 14개에서 일관되게 적용되어 있다.

**2D/3D 공존:**

하나의 템플릿으로 2D DOM Mixin과 3D Mixin을 모두 수용한다. 3D Mixin(MeshState, CameraFocus)의 모범답안이 "해당 없음" 처리 방식의 선례를 보여주고 있다.

**renderData 패턴과 직접 호출 패턴의 공존:**

7개 Mixin은 renderData 패턴(구독 기반)을 사용하고, CameraFocusMixin은 직접 호출 패턴을 사용한다. 템플릿의 섹션 3이 두 패턴을 모두 수용한다.

### C. 완결성 — 충분함

**커버하는 것:**

| 항목 | 대응 섹션 |
|------|----------|
| 존재 이유와 중복 방지 | 섹션 1 (기능 정의 + 기존 Mixin과의 관계) |
| 입력 인터페이스 | 섹션 2 (cssSelectors, datasetAttrs, 기타 옵션) |
| 데이터 계약 | 섹션 3 (renderData 기대 데이터) |
| 출력 인터페이스 | 섹션 4 (주입 네임스페이스) |
| 정리 범위 | 섹션 5 (destroy) |
| 사용 방법 | 섹션 6 (사용 예시) |

기능 모듈을 구현하기 위해 필요한 정보가 모두 있다.

**보완 가능 사항 (선택):**

| 항목 | 현재 상태 | 심각도 |
|------|----------|--------|
| 에러 조건 | 명세서에 없음. 설계 문서에 원칙("throw/catch")만 있음 | 낮음 |
| beforeDestroy.js 예시 | 명세서에 없음. 설계 문서에 범용 패턴 있음 | 낮음 |
| MeshState 괄호 보충 가이드 | 암묵적 허용 | 낮음 |

모두 심각도가 낮다. 설계 문서에서 이미 원칙을 다루고 있어, 명세서에 반복하지 않아도 구현에 지장이 없다.

## 최종 판정

### 구조 전략

**합당하다.** 기능(Mixin)과 시각(Design)의 직교 분리가 실제로 작동하며, N + M 관리량으로 N × M 조합을 달성하는 구조다. 대량 컴포넌트 확보 전략의 기반으로 적합하다.

### 명세 도구

**공식 배포 가능한 수준이다.** 일관성, 합리성, 완결성 모두 높다. 팀에게 이 템플릿 기반으로 명세를 작성해달라고 요청할 수 있다.
