합당한 부분: 개념 정리 → 코드 반영 방향, 용어 정확성, 감사 도구화
이건 이미 말했으니 반복하지 않겠습니다.

의문 1: 검증 없는 정제가 계속되고 있는가?

MEMORY.md (2026-02-07):
  "핵심 병목: [A] SKILL 선택이 아니라 [B] 코드 생성 정확도"
  "다음 단계: 실제 SKILL로 컴포넌트 생성 테스트 → 실패 지점 관찰 → 개선"

오늘(03-28) 10개 커밋: 전부 문서 정합성/용어/패턴 정리입니다.

질문: 이 정제가 실제로 코드 생성 정확도를 개선하는지
검증한 적이 있습니까?

문서를 아무리 정확하게 만들어도, SKILL이 그 정확함을 활용하지
못하면 효과가 없습니다. "다음 단계"가 2달째 다음 단계인지,
아니면 이미 테스트하고 결과를 반영 중인지에 따라
이 작업의 가치가 달라집니다.

의문 2: 관심사 분리가 소비자에게 유리한가?

이 문서의 소비자는 두 부류입니다:
  A) 사람 개발자 — register.js를 작성할 때 참조
  B) SKILL (AI) — 컴포넌트 코드를 생성할 때 참조

현재 구조:
  cssSelectors 섹션  → "container, template이 필수"
  itemKey 섹션       → "이 값은 cssSelectors에 있어야 함" (여기서 제약 설명)
  datasetAttrs 섹션  → "cssSelectors와 key 공유"
  사용 예시 섹션     → 전체를 조합한 코드

관심사 분리는 개념적으로 깔끔합니다.
하지만 소비자 입장에서는 세 섹션을 읽고 머릿속에서 조합해야 합니다.

특히 SKILL이 코드를 생성할 때:
  "cssSelectors에 넣을 키 결정 → 그 중 datasetAttrs에 넣을 키 결정
   → 그 중 itemKey로 쓸 키 결정 → 제약 조건 충족 확인"

이 합성 과정에서 실수가 생길 수 있습니다.
오늘 제가 {itemKey}를 cssSelectors 테이블에 넣은 것도
이 합성을 잘못한 사례입니다.

관심사 분리가 맞는 원칙이라 해도, 소비자가 AI라면
"분리된 원칙을 합성"하는 것보다
"통합된 예시를 따라하는 것"이 더 정확할 수 있습니다.

의문 3: 감사가 표면을 잡고 있는가, 구조를 잡고 있는가?

감사 도구가 찾은 것:
  - SystemInfo의 status cssSelector 누락 (실제 버그) ✅
  - 문서 링크 경로 오류
  - 폐기 용어 잔재
  - 컴포넌트 목록 불일치

감사 도구가 못 찾은 것:
  - ack가 datasetAttrs에만 있고 cssSelectors에 없는 패턴 위반
    (오늘 CTO 리뷰에서 발견)
  - {itemKey}가 cssSelectors 인터페이스 테이블에 들어간 관심사 위반
    (지금 직접 발견)

표면적 정합성(링크, 용어, 목록)은 잘 잡지만,
설계 수준 정합성(패턴 위반, 관심사 침범)은 잡지 못했습니다.
감사 도구의 점검 깊이가 충분한지 재검토할 여지가 있습니다.

종합:

원칙 자체는 합당합니다.
방향도 논리적입니다.

단, "이 정제가 실제 목표(코드 생성 정확도)에
기여하는지"를 검증하지 않으면,
정확하지만 효과 없는 문서를 만들고 있을 가능성이 있습니다.
