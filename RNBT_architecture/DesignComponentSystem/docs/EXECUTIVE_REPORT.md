# 기능이 있는 디자인 컴포넌트 — 문제와 해결 방향

## 문제

RENOBIT은 웹 콘텐츠 플랫폼이다. 어떤 프로젝트가 진행될지, 어떤 콘텐츠가 제작될지 사전에 알 수 없다.

이 상황에서 **"기능이 있는 디자인 컴포넌트를 만들자"**라는 요구가 있다.

콘텐츠는 프로젝트마다 달라지므로, 콘텐츠를 기준으로 삼을 수 없다. 그러면 '기능'이란 무엇인가? — 이 정의가 없으면 무엇을 만들어야 하는지 알 수 없고, 작업을 반복할 수 없다.

---

## 기능을 어떻게 정의할 것인가

컴포넌트를 만들겠다고 고민을 시작하고, 머리속에 무엇을 만들지 떠올려보자. 네비게이션, 트리, 버튼, 이미지 — 이런 것들이 혹시 떠오르는가? 아니면 데이터 타입에 따른 콘텐츠가 떠오를 수도 있다.

그러나 이것들은 실제로는 시각적 형태, 또는 데이터 타입이지, 기능이 아니다. 형태를 기능처럼 다루면 "테이블 컴포넌트에 뭐가 있어야 하지?", "트리 컴포넌트에 뭐가 들어가야 하지?" 라는 도돌이표 같은 질문이 계속 반복된다. 데이터 타입으로 분류하면 데이터 타입 개수만큼 기능 목록이 끝없이 증가하게 된다.

목적을 더 올려보면 다른 것이 보인다. 테이블이라고 부르는 것에 대한 요구의 본질은 **"데이터를 받아서 요소를 생성하여 표시한다"**이다. 이 수준에서 보면 테이블도, 목록도, 카드도, 차트도 같은 목적의 변형이다.

데이터 타입도 마찬가지다. "UPS 운전모드를 표시한다", "배터리 전압을 표시한다", "냉각수 온도를 표시한다" — 데이터 항목은 다르지만, 하는 일의 본질은 모두 **"데이터 값을 화면 요소에 표시한다"**이다.

그런데 "데이터를 보여준다"는 **목적**이지, 아직 **기능**이 아니다.

### 목적과 기능의 경계

일상의 사례로 확인할 수 있다. "전화 기능"이라고 할 때, "의사소통한다"만으로는 기능이라 부르지 않는다. 이것은 목적이다.

"어떻게 의사소통하는가?"라고 물으면 — 전화로, 편지로, 이메일로 — 여러 답이 나온다. **"어떻게?"에 대한 답이 복수로 나오면, 그것은 아직 기능이 아니라 목적이다.**

"전화 기능"이라고 말하는 순간, 수단은 "음성 통화"로 특정된다. **"어떻게?"에 대한 답이 하나로 특정되면, 그것이 기능이다.** 선택이 이루어진 상태이다.

| 기능 | 보편화된 목적 | 특수한 상황 (수단) | 결합 |
|------|-------------|-------------------|------|
| 전화 기능 | 의사소통한다 | 원거리 + 음성 | 음성으로 원거리 통화한다 |
| 청소 기능 | 깨끗하게 만든다 | 바닥 + 진공 흡입 | 진공으로 바닥의 먼지를 제거한다 |
| 보관 기능 | 물건을 유지한다 | 공간 + 출납 | 공간에 물건을 넣어두고 꺼낸다 |

기능은 **보편화된 목적(행위)을 특수한 상황과 결합한 단위**이다.

### 이 정의를 시스템에 적용하면

"데이터를 보여준다"는 목적이다. "어떻게?"라고 물으면 여러 답이 나온다. 각 답이 하나의 기능이 된다:

| 기능 | 목적 | "어떻게?"에 대한 답 (= 수단) |
|------|------|---------------------------|
| FieldRenderMixin | 데이터를 보여준다 | 기존 DOM 요소의 텍스트에 필드값을 넣는다 |
| ListRenderMixin | 데이터를 보여준다 | template을 복제하여 반복 생성한다 |
| StatefulListRenderMixin | 데이터를 보여준다 + 상태를 반영한다 | template 복제로 반복 생성 + 개별 항목의 상태 변경 |
| EChartsMixin | 데이터를 보여준다 | ECharts 인스턴스로 시각화한다 |
| TabulatorMixin | 데이터를 보여준다 | Tabulator 인스턴스로 테이블을 구성한다 |
| HeatmapJsMixin | 데이터를 보여준다 | heatmap.js로 열 분포를 시각화한다 |
| MeshStateMixin | 상태를 반영한다 | 3D 메시의 색상을 데이터에 따라 변경한다 |
| CameraFocusMixin | 시점을 이동한다 | 카메라를 특정 대상으로 애니메이션 이동시킨다 |

같은 목적("데이터를 보여준다") 아래에서 수단이 다르면 서로 다른 기능이 된다. 이것은 "전화 기능"과 "이메일 기능"이 같은 목적("의사소통한다") 아래에서 서로 다른 기능인 것과 같다.

이제 기능이 정의되었다. 그러면 이 기능이 어떻게 "기능이 있는 디자인 컴포넌트"가 되는가?

## 다음 문제: 기능과 디자인이 하나로 묶여 있다

"기능이 있는 디자인 컴포넌트"를 분석하면, 이것은 기능과 디자인이 모두 포함된 하나의 덩어리를 의미한다.

기능과 디자인이 분리되지 않으면 다음 문제가 생긴다.

기능을 재사용할 수 없다. 기능 코드가 디자인의 내부 구조를 직접 참조하고 있으므로, 디자인이 바뀌면 기능도 다시 작성해야 한다. 예를 들어, 데이터를 보여줘야 하는 태그가 사라지고 다른 태그가 들어오면 기능 코드를 재검토해야 한다.

컴포넌트 수 = 관리량이 된다. 컴포넌트 100개면 기능 코드도 100벌이다.

기능과 디자인을 분리해야 한다. 그런데 어떻게 분리하는가? 단순히 파일을 나누는 것만으로는 해결되지 않는다. 기능이 디자인의 구조를 알아야 동작하고, 디자인이 기능의 존재를 알아야 결합되기 때문이다.

빨간 전화와 파란 전화를 떠올려보면, 색이 달라도 전화 기능은 동일하다. 색은 기능을 구성하지 않는다. 마찬가지로 디자인(시각 구조)이 달라도 기능은 동일할 수 있다. 이 분리를 실현하는 것이 인터페이스이다.

---

## 해결: 인터페이스가 분리를 가능하게 한다

디자인과 기능 사이에 **인터페이스(약속된 접점)**를 둔다.

```
디자인(HTML)                      기능 모듈
   │                                │
   │  "나는 이 접점들을             │  "그 접점에
   │   가지고 있다"                 │   이 행위를 붙인다"
   │                                │
   └────────── 인터페이스 ──────────┘
               (약속된 접점)
```

디자인은 약속된 접점만 유지하면, 나머지는 자유롭게 바꿀 수 있다.

기능은 접점만 알면 되고, 디자인의 내부 구조를 알 필요가 없다.

디자인이 바뀌어도 기능이 안 깨진다. 기능이 추가되어도 디자인을 수정할 필요가 없다.

이 인터페이스가 디자인과 기능의 독립성을 실제로 보장하는 핵심이다.

```
빨간 전화 / 파란 전화        →  디자인 (시각 구조가 다름)
전화 기능                    →  기능 모듈 (기능은 동일)
```

실제 구현(ListRenderMixin)에서 이 구조가 어떻게 동작하는지 보면:

```
기능(ListRenderMixin)이 정의한 KEY     디자인(HTML)이 제공하는 VALUE
─────────────────────────────────     ──────────────────────────
container                         →   '.event-log__list'
item                              →   '.event-log__item'
template                          →   '#event-log-item-template'
time                              →   '.event-log__time'
message                           →   '.event-log__message'
```

실제 코드에서는 다음과 같이 매핑된다:

```javascript
applyListRenderMixin(this, {
    cssSelectors: {
        container: '.event-log__list',           // 항목이 추가될 부모
        item:      '.event-log__item',           // 각 항목의 루트
        template:  '#event-log-item-template',   // 복제 대상
        time:      '.event-log__time',           // 시간 표시
        message:   '.event-log__message'         // 메시지 표시
    },
    datasetAttrs: {
        level:     'level'                       // HTML의 data-level 속성
    }
});
```

cssSelectors의 KEY는 두 종류다. `container`와 `template`은 Mixin이 규약으로 요구하는 KEY로, Mixin 내부에서 직접 참조한다. 나머지 KEY(`item`, `time`, `message` 등)는 사용자 정의 KEY로, Mixin은 이를 개별적으로 알지 못하며 일괄 순회하여 데이터를 반영한다. 둘 다 cssSelectors라는 동일한 인터페이스 안에 있으며, 디자인이 `.event-log__list`를 `.alarm-list`로 바꿔도 VALUE만 변경하면 기능 코드는 그대로다.

## 이 구조가 만드는 결과

### 1. 관리량은 더하기, 생산량은 곱하기

디자인과 기능이 인터페이스로 독립되어 있으므로, 자유롭게 조합할 수 있다.

디자인 N종과 기능 M종을 각각 관리하면 관리 대상은 N + M개다. 그러나 이 N + M개로 만들 수 있는 조합은 N × M 이상이다. 하나의 디자인에 여러 기능을 동시에 조립할 수 있으므로 경우의 수는 더 늘어난다.

기능을 개별 컴포넌트마다 직접 구현하면 N × M개를 각각 만들어야 한다. 이 구조에서는 N + M개를 관리하면 된다.

### 2. 디자인 추가 시 기능 개발이 불필요하다

새로운 디자인이 확정되면, 약속된 접점만 포함시키고 기존 기능 모듈을 그대로 적용한다. 기능 코드를 다시 작성할 필요가 없다.

디자인이 나오는 속도가 곧 컴포넌트 생산 속도가 된다.

### 3. 기능 추가 시 기존 디자인 전체에 적용 가능하다

새로운 기능 모듈을 1개 추가하면, 기존 모든 디자인에 해당 기능을 적용할 수 있다.

기능 1개의 추가가 전체 컴포넌트 카탈로그의 확장으로 이어진다.

### 4. 특수한 요구도 수용할 수 있다

프로젝트 고유의 특수한 요구는 프로젝트 단위에서 대응한다. 사전에 모든 것을 준비하려고 하지 않는다. 프로젝트에서 대응한 기능도 동일한 구조(목적 + 수단)를 가지며, 아직 기능 모듈로 추출하는 작업이 되지 않았을 뿐이다. 작업이 진행되면 시스템이 성장한다.

## 정리

| 질문 | 답 |
|------|-----|
| '기능'이란 무엇인가? | 보편화된 목적(행위)을 특수한 상황(수단)과 결합한 단위 |
| 목적과 기능은 어떻게 구분하는가? | "어떻게?"에 대한 답이 복수면 목적, 하나로 특정되면 기능 |
| 보편적 기능을 어떻게 발견하는가? | 구체적 요구의 목적을 추상화한다 |
| 기능을 디자인에 어떻게 붙이는가? | 약속된 접점(인터페이스)으로 결합한다 |
| 왜 인터페이스가 핵심인가? | 디자인과 기능이 서로를 몰라도 결합/교체/추가가 가능해진다 |
| 관리 부담은? | N + M (더하기) |
| 생산 가능량은? | N × M 이상 (곱하기 + 조합) |
| 디자인이 추가되면? | 기존 기능 즉시 적용, 기능 개발 불필요 |
| 기능이 추가되면? | 기존 디자인 전체에 적용 가능 |
| 예측 못한 요구는? | 프로젝트에서 대응 — 동일한 구조의 기능이며, 기능 모듈로의 추출은 작업 진행에 따른다 |
