# HANA BANK HIT Dev — 디자이너 경향(Design Tendency)

> 이 문서는 **토큰 사전이 아니다.** 8개 컴포넌트 샘플(`cpu-status`, `event-browser`, `institution-delay-top5`, `performance-monitoring`, `performance-status`, `performance-status-v2`, `server-icon2`, `time-trend-chart`)에서 관찰된 이 디자이너의 **사고 패턴**을 기록한 것이다.
>
> 목적: Figma 원본 없이 새 컴포넌트를 요청받았을 때, "이 디자이너라면 어떻게 결정했을까"를 추론하여 **동일한 손맛의 결과물**을 생성하는 것.
>
> 사용법: 새 컴포넌트 생성 전 이 문서를 읽고, 아래의 "의사결정 휴리스틱"과 "피하는 것"을 판단 기준으로 삼는다. 값을 그대로 복사하지 말고, **원칙에서 값을 도출**하라.

---

## 한 줄 정의

**금융 기관 관제실을 위한, 네온 한 점만 허용하는 절제된 다크 모니터링 미학.**

신뢰·정밀·긴장감을 동시에 전달해야 하는 환경에서, 장식을 거의 쓰지 않고 **타이포그래피와 상태 색상만으로** 정보 위계를 구축한다.

---

## 핵심 철학 5가지

### 1. "강조색은 단 하나의 축이다"
터콰이즈/사이안 계열 한 축 외에는 강조색을 쓰지 않는다. 보라·핑크·파랑을 섞지 않는다. 다양성은 **상태 색상(정상/주의/경계/심각) 4단계**에서만 허용된다.

### 2. "검정에도 감정이 있다"
배경은 순흑(`#000`)이 아니다. 미세한 그린 틴트가 섞인 검정을 선호한다. 두 겹의 그라데이션(검정 레이어 + 그린 톤 레이어)으로 깊이를 만든다. 평면적인 단색 배경을 피한다.

### 3. "데이터가 말하고 라벨은 속삭인다"
큰 메트릭 숫자는 **얇게(Light 300)** 쓰고, 작은 라벨은 **굵게(Medium~SemiBold)** 쓴다. 크기 대비가 아니라 **무게의 역전**으로 위계를 만든다. 숫자가 주인공이고 라벨은 그 그림자다.

### 4. "있는 듯 없는 경계"
테두리는 서브픽셀(0.3px~0.75px)로 가늘게 긋는다. 섀도우는 외곽 드롭섀도우 대신 **인셋 섀도우(낮은 opacity)**로 내부 깊이만 암시한다. 경계선이 "여기에 있다"고 외치지 않고, 스캔할 때 무의식적으로만 지각되게 한다.

### 5. "픽셀은 반올림하지 않는다"
10.108px, 8.086px, 0.248px 같은 소수점 값이 등장한다. Figma 원본 정밀도를 의도적으로 **보존**한다. 반올림으로 "깔끔하게" 만드는 관행을 거부한다. 이는 완벽주의의 흔적이다.

## 의사결정 휴리스틱

새 컴포넌트를 만들 때 각 영역에서 내려야 할 결정들. 값이 아니라 **판단 기준**이다.

### 색상 결정

| 상황 | 판단 기준 |
|------|-----------|
| 배경색을 고를 때 | 순흑 금지. 진한 녹-검정 계열에서 선택. 카드는 반투명으로 두 층을 겹친다. |
| 강조가 필요할 때 | 터콰이즈/사이안 축에서만 선택. "좀 더 튀게 만들고 싶다"는 욕구를 새 색상 추가로 해결하지 않는다. |
| 상태를 구분할 때 | 반드시 4단계(정상·주의·경계·심각). 2단계로 뭉뚱그리지 않는다. "그냥 빨강/초록"은 이 디자이너의 어휘가 아니다. |
| 텍스트 색을 정할 때 | 순백(`#fff`) 금지. 미세한 그린/웜 틴트가 섞인 밝은 회색. |
| 채도 전략 | 강조색은 네온급 고채도, 배경·텍스트는 저채도. **중간 채도가 가장 적다** — 극단 대비를 선호. |

### 타이포 결정

| 상황 | 판단 기준 |
|------|-----------|
| 큰 숫자(메트릭)를 배치할 때 | 얇은 weight(Light 300), 큰 사이즈. "얇고 크다"가 이 디자이너의 시그니처. |
| 라벨·캡션을 쓸 때 | 작은 사이즈, Medium~SemiBold. 숫자와 반대 방향으로. |
| 제목을 쓸 때 | 18px/600 근방. 과하게 키우지 않는다. 아이콘과 9px 갭으로 짝을 이룬다. |
| 폰트 선택 | Pretendard 단일. 세리프·다른 산세리프 혼용 금지. |
| 자간 조정 | 한글은 거의 항상 음수 letter-spacing. 타이트한 밀도가 기본값. |
| 사이즈 스케일 확장 욕구 | 4~5단계로 충분하다. 8단계 스케일은 과하다 — weight로 해결한다. |

### 공간/형태 결정

| 상황 | 판단 기준 |
|------|-----------|
| 패딩 비율 | **가로가 세로의 약 1.5~2배**. 가로로 숨쉬는 리듬을 선호한다. |
| radius 단계 | 3단계 고정 감각: 카드(넉넉) / 버튼·뱃지(중간) / 인라인(타이트). pill 형태는 상태 배지에만 쓴다. |
| 정보 밀도 | 높게. 여백으로 숨 돌리기보다 **정렬과 분리선**으로 정돈한다. "더 비워야 할까?" 할 때 이 디자이너는 "더 정렬하자"로 간다. |
| 구분선 스타일 | 실선보다 dashed를, 두껍기보다 얇기를 선호. 색은 배경보다 약간만 밝은 저채도 그린. |
| 그리드 | Flexbox가 기본, 겹침이 필요할 때만 Grid의 같은 셀 오버레이를 쓴다. |

### 효과 결정

| 상황 | 판단 기준 |
|------|-----------|
| 강조가 필요할 때 | drop shadow ❌ / 얕은 **glow** ⭕. 상태 배지는 색상별 glow로만 강조. |
| 깊이감을 줄 때 | 외곽 섀도우 ❌ / **인셋 섀도우(opacity 0.06 근방)** ⭕. 카드는 안쪽으로 약간 팬다. |
| 블러 사용 | 있다, 그러나 매우 약하게(2px 전후). 유리 효과가 아니라 "살짝 물러남"의 암시. |
| 그라데이션 방향 | 색상 전환이 아니라 **톤 전환**. 검정 → 검정-그린처럼 같은 계열 안에서. 무지개/파스텔 그라데이션 금지. |
| 애니메이션 | 관찰된 샘플에서 강한 모션 없음. 있다면 상태 전환의 짧은 페이드 정도로 그친다(추론). |

### 컴포지션 결정

| 상황 | 판단 기준 |
|------|-----------|
| 컴포넌트 골격 | **헤더 + 본문** 2-part 구조가 기본. 헤더는 예외 없이 존재. |
| 헤더 구성 | SVG 아이콘 + 제목 텍스트, gap 9px, 왼쪽 정렬. |
| 테이블 행 차별화 | 색상 변경 ❌ / **투명도 차이** ⭕. 같은 색을 0.3 / 0.7 두 단계로 zebra. |
| 버튼 상태 | hover 중심이 아니라 **active / inactive 2상태**가 명확. 관제 UI라 "지금 선택된 것"이 더 중요하다. |
| 아이콘 자산 | 비트맵 ❌ / **SVG** ⭕. 크기는 컴포넌트 맥락에 맞춰 소수점까지. |
| 진행 바 | 배경은 저채도 밝은 회색, 채우기는 터콰이즈. 높이는 얇게(10px 이하). |

## 피하는 것 (Negative Rules)

이 디자이너가 **거의 확실히 거부할** 선택들. 생성 중 이 패턴들이 나타나면 멈추고 재검토하라.

### 색상
- ❌ 라이트 모드, 화이트 배경
- ❌ 보라 + 파랑 + 핑크 같은 다중 강조색
- ❌ 파스텔, 베이비 톤
- ❌ 순흑 `#000000` 또는 순백 `#ffffff`
- ❌ 빨강/초록 2단계 상태(주의·경계 단계를 생략)

### 타이포
- ❌ 세리프 폰트 혼용
- ❌ 숫자에 굵은 weight, 라벨에 얇은 weight (역방향)
- ❌ 양수 letter-spacing(한글에서 자간 벌리기)
- ❌ 8단계 이상의 과한 사이즈 스케일

### 공간/형태
- ❌ 픽셀 반올림 강박("10.108px는 10px로 바꿔야지" → 금지)
- ❌ 세로 패딩이 가로보다 큰 박스
- ❌ radius 과용(모든 요소를 pill로 만드는 경향)
- ❌ 두꺼운 실선 구분(1.5px 이상)

### 효과
- ❌ Material Design 스타일의 큰 drop shadow(elevation)
- ❌ 고채도 유색 섀도우(보라 섀도우 등)
- ❌ 강한 배경 블러(glassmorphism 전면 사용)
- ❌ 무지개/홀로그램 그라데이션
- ❌ 과장된 hover 변환(scale, 큰 이동)

### 컴포지션
- ❌ 헤더 없는 컴포넌트
- ❌ 행 구분을 색상 hue 변경으로 하는 zebra
- ❌ 3상태 이상의 복잡한 버튼(primary/secondary/tertiary/ghost…) — 2상태로 충분하다고 본다

## 적용 체크리스트 (새 컴포넌트 생성 시)

생성 후 다음 질문에 모두 "예"로 답할 수 있는지 검증하라.

- [ ] 배경이 순흑이 아니라 미세한 그린 틴트가 섞인 검정인가?
- [ ] 강조색이 터콰이즈/사이안 축 **한 가지**인가?
- [ ] 상태 표현이 4단계 어휘를 따르는가? (해당되는 경우)
- [ ] 큰 숫자는 얇고(Light), 작은 라벨은 굵은가?
- [ ] Pretendard 단일 폰트인가?
- [ ] 한글 자간이 음수인가?
- [ ] 가로 패딩이 세로 패딩보다 넓은가?
- [ ] 테두리가 서브픽셀(< 1px)인가?
- [ ] 외곽 drop shadow 대신 인셋 섀도우 또는 glow를 쓰는가?
- [ ] 헤더(아이콘 + 제목)가 존재하는가?
- [ ] 행/리스트의 차별화가 색상이 아닌 투명도 차이인가?
- [ ] 아이콘이 SVG인가?
- [ ] "피하는 것" 목록의 어느 항목도 등장하지 않는가?

## 관찰 근거 (Evidence Map)

각 경향은 다음 샘플에서 관찰되었다. 새 경향을 추가하거나 기존 경향을 수정할 때 근거 샘플을 함께 기록한다.

| 경향 | 근거 샘플 |
|------|-----------|
| 검정-그린 이중 레이어 배경 | `performance-status`, `performance-monitoring`, `institution-delay-top5` |
| Light 300 큰 숫자 | `cpu-status`, `performance-status-v2` |
| 상태 4단계 어휘 | `event-browser`, `performance-monitoring` |
| 서브픽셀 테두리 | 거의 모든 샘플 |
| 인셋 섀도우 | `cpu-status`, `time-trend-chart` |
| Dashed 세로 구분선 | `event-browser`, `institution-delay-top5` |
| 투명도 기반 zebra | `event-browser`, `institution-delay-top5` |
| 소수점 픽셀 | `performance-status-v2`(10.108px, 8.086px) 외 다수 |
| 2상태 버튼 | `event-browser` |

## 문서 갱신 원칙

- 이 문서는 **경향**을 기록한다. 특정 값(hex, px)은 예시일 뿐이며, 신뢰해야 할 것은 **판단 기준**이다.
- 새 샘플이 추가되면, 기존 경향을 **강화/수정/반증**하는지 검토하고 반영한다.
- 한 사례만으로 경향을 단정하지 않는다. 최소 2개 이상의 샘플에서 반복될 때 경향으로 승격한다.
- 값 복사가 필요할 땐 원본 CSS를 직접 읽는다. 이 문서는 **"왜 그렇게 만들었는가"**의 기록이다.
