# 하네스 엔지니어링 관점의 WEB_BUILDER_TOOLKIT 평가

> **기준 문서:** [harness-engineering.md](harness-engineering.md) (OpenAI, Ryan Lopopolo, 2026-02-11)
> **평가일:** 2026-03-29

---

## 평가 프레임워크

하네스 엔지니어링의 핵심 명제:

> "사람은 조종하고, 에이전트가 실행한다."
> → 엔지니어의 핵심 업무 = 환경 설계 + 의도 명세 + 피드백 루프 구축

이 프레임워크의 **7개 축**으로 WEB_BUILDER_TOOLKIT을 평가한다.

---

## 1. 지식 구조화 — ★★★★☆

> "에이전트에게 지도를 줘라, 1,000페이지 설명서 말고."

### 하네스 엔지니어링 원칙

- AGENTS.md는 목차 역할, 지식은 구조화된 docs/에 배치
- 점진적 공개 (progressive disclosure)
- 단일 거대 파일 → 실패

### 현황

**잘 되어 있는 점:**

- ✅ 3-tier CLAUDE.md 구조 (Root → Figma_Conversion → RNBT_architecture)
  - Root CLAUDE.md가 목차 + SKILL 선택 가이드 역할
  - 각 하위 CLAUDE.md는 해당 도메인의 원칙만 정의
- ✅ 참조 문서 분리: guides/ (CODING_STYLE, FIGMA_MCP_GUIDE 등)
- ✅ SHARED_INSTRUCTIONS.md로 공통 규칙 단일 출처화 (커밋 `5a349ad`)
- ✅ SKILL이 참조 링크로 연결 → 중복 제거 완료

**개선 여지:**

- △ Root CLAUDE.md가 목차 이상의 역할을 함 (검증 원칙, 작업 태도 등)
  - 하네스 엔지니어링: "진입점은 작고 안정적"이어야 함
  - 작업 원칙/태도는 별도 문서로 분리하면 진입점이 더 깔끔해짐
- △ 설계 문서(design-docs), 실행 계획(exec-plans) 개념이 부재
  - 하네스 엔지니어링: "계획은 일급 아티팩트"
  - 현재는 SKILL이 실행 계획을 암묵적으로 대체하지만, 복잡한 작업의 의사결정 이력이 리포지토리에 남지 않음

## 2. 기계적 강제 — ★★☆☆☆

> "문서가 부족하면, 규칙을 코드로 승격시킨다."

### 하네스 엔지니어링 원칙

- 커스텀 린터 + 구조적 테스트로 불변 조건 강제
- 에러 메시지에 수정 지침을 넣어 에이전트 컨텍스트에 주입
- "한 번 인코딩되면 모든 곳에 동시에 적용"

### 현황

**잘 되어 있는 점:**

- ✅ Hook 전략 문서가 이미 존재 ([HOOKS_FOR_CONSISTENCY.md](.claude/docs/HOOKS_FOR_CONSISTENCY.md))
  - 4개 패턴에 대한 PostToolUse Hook 설계 완료
  - Phase 1(피드백) → Phase 2(차단) 점진적 도입 계획
- ✅ Hook의 한계를 정직하게 인식 ("정규식으로 표현할 수 없는 대부분의 규칙")

**핵심 갭:**

- ✖ Hook이 아직 미구현 (설계 문서만 존재, 실제 배포 없음)
  - 하네스 엔지니어링: "문서가 부족하면 규칙을 코드로 승격"
  - 현재: 20건의 불일치를 발견하고 설계까지 했으나 실행 미완
- ✖ CI/린터/구조적 테스트 부재
  - 하네스 엔지니어링은 커스텀 린터가 교차 링크, 구조, 신선도를 검증
  - WEB_BUILDER_TOOLKIT: 강제 메커니즘 = 텍스트 규칙 + 사람의 감사
- ✖ 에러 메시지 → 에이전트 컨텍스트 주입 패턴 미활용
  - Hook 설계에는 반영되어 있으나 (stderr 피드백), 실제 작동하지 않음

> **이 축이 가장 큰 격차다.**
> 하네스 엔지니어링의 핵심 통찰은 "텍스트 규칙 → 코드 강제"인데,
> WEB_BUILDER_TOOLKIT은 이 전환의 설계 단계에서 멈춰 있다.

## 3. 에이전트 가독성 — ★★★★☆

> "접근할 수 없는 것은 사실상 존재하지 않는다."

### 하네스 엔지니어링 원칙

- 리포지토리 로컬의 버전 관리된 아티팩트만이 에이전트가 볼 수 있는 전부
- Slack 논의, Google Docs → 에이전트에게 불가독
- "지루한" 기술이 에이전트가 모델링하기 쉬운 경향

### 현황

**잘 되어 있는 점:**

- ✅ 모든 지식이 리포지토리 내 마크다운으로 존재
  - CLAUDE.md, guides/, docs/, SKILL.md, README.md
  - 외부 시스템 의존 없음 (Linear, Slack 등 참조 없음)
- ✅ SKILL 파일이 완전한 자기 설명적 구조
  - description, instructions, 입출력 명세가 SKILL.md 안에 완비
- ✅ 기술 스택이 "지루한 기술"에 해당
  - 바닐라 JS, fx.js (자체 함수형 라이브러리), CSS
  - 에이전트가 모델링하기 쉬운 단순한 추상화
- ✅ Figma MCP 통합으로 디자인 정보도 에이전트가 직접 접근 가능

**개선 여지:**

- △ preview_runtime.js의 런타임 시뮬레이션이 있지만, 에이전트가 실행 결과를 직접 관찰하는 피드백 루프는 제한적
  - 하네스 엔지니어링: Chrome DevTools Protocol로 에이전트가 DOM 스냅샷, 스크린샷, 네비게이션 직접 수행
  - WEB_BUILDER_TOOLKIT: Playwright 스크린샷 비교는 수동 트리거

## 4. 계층적 아키텍처 — ★★★☆☆

> "Types → Config → Repo → Service → Runtime → UI"

### 하네스 엔지니어링 원칙

- 각 도메인이 고정된 레이어 세트로 나뉨
- 엄격하게 검증된 의존성 방향
- 교차 관심사는 단일 인터페이스(Providers)로 진입
- 코딩 에이전트와 함께라면 "초기 전제 조건"

### 현황

**잘 되어 있는 점:**

- ✅ 2단계 파이프라인이 명확한 레이어링
  - Figma_Conversion (정적 퍼블리싱) → RNBT_architecture (동적 런타임)
- ✅ SKILL이 5단계로 계층화 (1-figma → 2-component → 3-page → 4-review → 5-mixin)
- ✅ Mixin 시스템이 교차 관심사를 구조화 (FieldRenderMixin, ListRenderMixin, EChartsMixin 등 9종)
- ✅ register.js의 3단계 구조 (Mixin 적용 → 구독 연결 → 이벤트 매핑)

**핵심 갭:**

- ✖ 의존성 방향의 기계적 강제 없음
  - 하네스 엔지니어링: 의존성 방향을 린터가 검증
  - WEB_BUILDER_TOOLKIT: 텍스트 문서에 "이렇게 해야 한다"만 존재
- ✖ 레이어 간 인터페이스가 암묵적
  - Mixin이 교차 관심사를 담당하지만, 어떤 Mixin이 어떤 컴포넌트에서 사용 가능한지의 제약이 문서적
  - "Providers" 같은 단일 진입점 개념 없음

## 5. 피드백 루프 — ★★☆☆☆

> "교정은 저렴하고, 기다리는 것은 비싸다."

### 하네스 엔지니어링 원칙

- 앱을 Git worktree별로 부팅하여 변경마다 검증
- 에이전트가 버그 재현 → 수정 → 영상 녹화 → PR → 피드백 응답
- 단일 실행이 6시간+ 하나의 작업에 매달림

### 현황

**존재하는 피드백 루프:**

- ✅ Figma MCP → get_screenshot → Playwright 스크린샷 비교 ("시각적 차이" 감지)
- ✅ SKILL의 "작업 전 필수 확인" 패턴 (작업 시작 전 참조 문서 읽기를 SKILL이 지시)
- ✅ audit-project SKILL이 프로젝트 전체 진단 수행

**핵심 갭:**

- ✖ 자동화된 피드백 루프가 없음
  - 스크린샷 비교가 수동 트리거
  - 에이전트가 자율적으로 "수정 → 검증 → 재수정" 루프를 돌 수 없음
- ✖ audit-project가 수동 호출
  - 하네스 엔지니어링: 정기적 백그라운드 에이전트가 위반 스캔
  - WEB_BUILDER_TOOLKIT: 사용자가 /audit-project를 직접 호출해야 함
- ✖ 에이전트-대-에이전트 리뷰 없음
  - 하네스 엔지니어링: PR 리뷰를 에이전트가 에이전트에게 위임
  - WEB_BUILDER_TOOLKIT: 모든 리뷰가 사람 의존

## 7. 자율성의 단계적 확대 — ★★★☆☆

> "에이전트가 엔드-투-엔드로 구동한다."

### 하네스 엔지니어링 원칙

- 단일 프롬프트 → 검증 → 재현 → 수정 → 검증 → PR → 머지
- 판단이 필요할 때만 사람에게 에스컬레이션

### 현황

**자율성 수준:**

- ✅ SKILL 시스템이 단계별 자율 실행을 가능하게 함
  - /figma-to-html: Figma MCP → HTML/CSS 생성 → 스크린샷 검증
  - /create-standard-component: 정적 HTML → 동적 컴포넌트 변환
  - /create-project: 여러 컴포넌트 조합 → 대시보드 생성
- ✅ SKILL 간 파이프라인이 엔드-투-엔드 워크플로우를 형성 (1-figma → 2-component → 3-page → 4-review)

**핵심 갭:**

- ✖ 각 SKILL이 독립 실행이지, 자율 체이닝되지 않음
  - 하네스 엔지니어링: 단일 프롬프트로 전체 흐름
  - WEB_BUILDER_TOOLKIT: 사용자가 각 SKILL을 순차 호출해야 함
- ✖ "절대 서두르지 않습니다" 원칙이 자율성과 긴장
  - 하네스 엔지니어링: "교정은 저렴하고, 기다리는 것은 비싸다"
  - WEB_BUILDER_TOOLKIT: 매 단계마다 사용자 승인 대기
  - 이것은 의도적 선택이지만, 자율성 확대의 천장이 됨

## 성숙도 스펙트럼에서의 위치

```
[텍스트 규칙만]──[구조화된 지식]──[기계적 강제]──[자동 피드백]──[자율 실행]
                      ▲
                 WEB_BUILDER_TOOLKIT
```

- **강점:** "구조화된 지식" 단계를 충실히 달성. 3-tier CLAUDE.md, SKILL 시스템, 참조 문서 단일 출처화는 많은 프로젝트가 도달하지 못하는 수준
- **약점:** "기계적 강제" 이후 단계로의 전환이 시작되지 않음. Hook 설계는 되어 있으나 배포 안 됨. 자동화된 피드백 루프와 정기 정리가 없음

## 맥락의 차이 — 직접 비교의 한계

하네스 엔지니어링과 WEB_BUILDER_TOOLKIT은 근본적으로 다른 맥락이다:

| | 하네스 엔지니어링 | WEB_BUILDER_TOOLKIT |
|---|---|---|
| **에이전트** | Codex (비동기, 6시간+ 자율 실행) | Claude Code (대화형, 동기) |
| **팀 규모** | 3→7명 엔지니어 | 1인 (추정) |
| **코드 규모** | 100만 줄 | 소규모 |
| **산출물** | 내부 제품 (수백 사용자) | 컴포넌트 변환 도구 |
| **핵심 제약** | 사람 시간/주의력 | 코드 생성 정확도 |
| **코드 작성 주체** | 100% 에이전트 | 에이전트 + 사람 검증 |

이 차이가 의미하는 것:

1. **"교정은 저렴" vs "정확성이 최우선"**
   - 하네스 엔지니어링: 빠르게 머지하고 나중에 교정
   - WEB_BUILDER_TOOLKIT: Figma 픽셀 단위 일치가 목표 → "서두르지 않기" 원칙은 이 맥락에서 합리적 선택

2. **비동기 자율 실행 vs 대화형 실행**
   - Codex는 6시간 자율 실행 가능 → 자동 피드백 루프 필수
   - Claude Code는 대화 기반 → 사람이 루프에 자연스럽게 포함 → 자동 피드백 루프의 필요성이 상대적으로 낮음

3. **100만 줄 vs 소규모**
   - 규모가 커질수록 기계적 강제의 ROI가 급증
   - 소규모에서는 "잘 구조화된 문서"가 적정 수준일 수 있음

## 실행 가능한 개선 — ROI 기준 우선순위

### 높은 ROI — 즉시 실행 가능

**1. Hook 4개 실제 배포** (설계 완료 상태)

- HOOKS_FOR_CONSISTENCY.md의 Phase 1 스크립트를 `.claude/hooks/`에 실제 생성 + `settings.json`에 등록
- 비용: 낮음 (설계 완료, 코드 작성만 남음)
